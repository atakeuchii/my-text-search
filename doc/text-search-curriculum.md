# 全文検索エンジン実装 — 2週間カリキュラム（v2 / 検索種拡張版）

Clojureで転置インデックスベースのミニ全文検索エンジンを実装し、Elasticsearch（Lucene）のコア原理を自分の手で再現する2週間の学習計画。前回実装した LSM-tree ストレージエンジン（`my-storage`）を永続化層として再利用する。

このv2は、Boolean検索・BM25・フレーズ検索の本線に加えて、**検索の種類を広げる**方向（ワイルドカード・ファジー・ハイライト・フィールドboost・ファセット）を本線に組み込み、最後に**品質評価**で締める構成。

---

## 前提と方針

| 項目 | 決定 |
|---|---|
| 実装言語 | Clojure |
| トークナイズ | 日本語 + N-gram（bi-gram中心、自作） |
| 永続化層 | 前回実装したLSM-treeストレージエンジン（`my-storage`）を再利用 |
| スコアリング | TF-IDF → BM25 → フィールドboost |
| マッチング（本線） | AND/OR検索、フレーズ検索、ワイルドカード、ファジー |
| 結果提示（本線） | ランキング、ハイライト、ファセット集計 |
| 品質保証 | precision/recall による評価 |
| データセット | 学習用の小規模サンプル（数百〜数千件）。実践投入（いしなび/これ飲も）は2週間後に別途検討 |

### 最終ゴール

「テキスト集合に対して、キーワード検索・ワイルドカード・ファジー・フレーズ検索ができ、BM25 + フィールドboost でランキングし、ハイライトとファセットで結果を提示し、その品質を precision/recall で測れるミニ検索エンジン」を CLI/REPL から操作できる状態にする。Elasticsearch のデフォルト検索（BM25 + bool query + match_phrase）に、実務でよく使う検索種と評価を足した縮小版。

### Elasticsearchとの対応（作るもの / 作らないもの）

| 機能 | Elasticsearchでの呼称 | 今回 |
|---|---|---|
| AND/OR検索 | bool query (must/should) | ◎ 本線 |
| ワイルドカード | wildcard query | ◎ 本線 |
| ファジー検索 | fuzzy query | ◎ 本線 |
| フレーズ検索 | match_phrase | ◎ 本線 |
| BM25スコアランキング | _score | ◎ 本線 |
| フィールドboost | fields / boost | ◎ 本線 |
| ハイライト | highlight | ◎ 本線 |
| ファセット・集計 | aggregations | ◎ 本線 |
| 検索品質評価 | rank_eval API | ◎ 本線 |
| 分散・レプリケーション | sharding / replica | ✕ 対象外 |
| REST API層 | _search API | ✕ 対象外（REPL/CLIで代替） |
| near real-time search | refresh | ✕ 対象外 |

### 設計上の一貫方針

- **正規化はindex時とquery時で必ず同じ関数を通す**。片方だけ正規化すると同じ語でもトークンが一致せずヒットしない。これが全機能を貫く不変条件。
- **ポスティングcodecは位置情報の余白を最初から残す**。Day3のエンコード設計時に「文書IDのdelta列 + 各文書の可変長ペイロード」レイアウトにしておき、TF（Day7）と位置（Day10）を後から最小改修で足せるようにする。codecを丸ごと書き直す手戻りを避けるための折衷。
- **Week1でマッチング（どの文書が当たるか）を出し切り、Week2でランキングと提示（どう並べ・見せ・測るか）を積む**。

## Week 1: トークナイズ 〜 マッチングの種類

ゴール: 「サンプル文書をインデックス化し、AND/OR・ワイルドカード・ファジーでヒット文書集合を返せる」状態。

### Day 1 — N-gramトークナイザの自作 ✓完了

やること
- bi-gram（2文字）を基本としたトークナイザを実装。`"日本酒"` → `["日本", "本酒"]`。
- 正規化（NFKC、全角/半角統一、ロケール非依存の小文字化）を前段に置く。記号・空白の除去はトークナイズ側で「セグメント境界」として扱う。
- uni-gram / bi-gram / tri-gram を切り替えられるようにし、N数ごとのトークン数・重複を計測。

学習ポイント
- なぜ日本語で形態素解析ではなくN-gramを選ぶと「検索漏れが減るが誤ヒットが増える」のか（例: 「京都」で「東京都」がヒットする問題）。
- N数の選定が精度と索引サイズに与える影響。

チェックポイント
- [ ] 任意の日本語文字列をbi-gramトークン列に変換できる
- [ ] 正規化のユニットテストが通る（全角数字、濁点、記号など）
- [ ] N数を変えたときのトークン数・重複を計測できる（`bench/` に計測スクリプト）

既知の制約
- 数字の桁区切り（`"1,000"` と `"1000"` がトークン不一致）は未対応。必要になれば正規化段でカンマを数字間から除去する。

### Day 2 — 転置インデックスをオンメモリで構築

やること
- ポスティングリスト（term → 出現文書IDの一覧）のデータ構造を設計。
- 文書追加API（`add-document`）を実装。文書ID採番、term抽出、ポスティングへの追加。
- まずはClojureの永続データ構造（`sorted-map` + `sorted-set` など）でインメモリ実装。
- 文書本体（forward側）も保持しておく（後日ハイライト・ファセットで使う）。

学習ポイント
- 転置インデックスの構造（term辞書 + ポスティングリスト）。
- Clojureの永続データ構造・トランスデューサーがインデックス構築とどう噛み合うか。

チェックポイント
- [ ] 複数文書を追加し、任意のtermからポスティングリストを取得できる
- [ ] term辞書のサイズ、平均ポスティング長を確認できる

---

### Day 3 — LSM-treeへの統合 + ポスティングcodec

やること
- ポスティングリストをLSM-treeのkey/valueにマッピング。key = term、value = エンコードしたポスティング。
- 文書IDのデルタエンコーディング + 可変長整数（varint）で圧縮フォーマットを実装。
- **位置情報が後から入る余地を残したレイアウト**にする（文書IDのdelta列 + 各文書の可変長ペイロード）。
- インメモリ実装をLSM永続化層に差し替え、書き込み→再起動→読み込みが通ることを確認。

学習ポイント
- LuceneのセグメントとLSM-treeのSSTableの発想の近さ（書き込み最適化 vs 読み込み最適化）。
- delta + varint がなぜポスティングを小さくできるのか（IDが昇順なので差分が小さい）。

チェックポイント
- [ ] termをキーにポスティングをLSM-treeへ永続化できる
- [ ] プロセス再起動後もインデックスから検索できる
- [ ] ポスティングのエンコード/デコードのラウンドトリップテストが通る

### Day 4 — AND/OR（Boolean）検索

やること
- クエリをトークナイズし、各termのポスティングリストを取得。
- AND = 積集合、OR = 和集合をソート済みリストのマージで実装（skip前提の走査）。
- REPLまたは簡易CLIで検索できるようにする。

学習ポイント
- ポスティングリストの積集合・和集合アルゴリズム。
- ソート済みリストのマージがなぜ高速か。

チェックポイント
- [ ] `A AND B`、`A OR B` が正しいヒット文書集合を返す
- [ ] N-gramに分解された検索語が正しくAND結合される（例: 「日本酒」→「日本」AND「本酒」）
- [ ] 「検索できる」状態に到達

---

### Day 5 — ワイルドカード検索（前方一致）

やること
- `日本*` のような前方一致クエリを、term辞書のprefix走査で解決。
- term辞書がソート済み（`sorted-map` / LSMのSSTableもソート済み）である性質を使い、prefix範囲のtermを列挙 → 各ポスティングをOR結合。
- 拾いすぎ（prefixに大量termがぶら下がる）への上限やコストの体感。

学習ポイント
- ワイルドカードが「マッチするtermをまず集めてからポスティングをOR」する多段構造であること。
- ソート済みterm辞書がprefix検索にそのまま効く理由。範囲スキャン（`my-storage` の `scan`）の再利用。

チェックポイント
- [ ] `日本*` が「日本」で始まるtermを含む文書を返す
- [ ] prefixにマッチするterm集合を列挙できる
- [ ] 中間・後方ワイルドカードは対象外と割り切れている（前方一致索引の限界を説明できる）

---

### Day 6 — ファジー検索（編集距離）

やること
- レーベンシュタイン距離を実装（DP、必要ならバンド化で高速化）。
- クエリtermと編集距離 ≤ k のtermをterm辞書から探し、マッチtermのポスティングをOR結合。
- 素朴な全term走査の遅さを体感し、prefix枝刈り等で現実的な速度にする。

学習ポイント
- レーベンシュタイン距離のDP構造。
- ファジー検索が「近いtermを集めてからOR」というワイルドカードと同型の多段構造であること。
- 閾値kが再現率と誤ヒットに与える影響（次のDay13評価への伏線）。

チェックポイント
- [ ] クエリと編集距離1〜2のtermにマッチする文書を返す
- [ ] タイポ入りクエリで正しい文書がヒットする
- [ ] kを変えたときのヒット数の変化を観察できる
- [ ] Week1完了: 4種のマッチング（AND/OR・ワイルドカード・ファジー）が動く

## Week 2: ランキング 〜 提示 〜 評価

ゴール: 「BM25 + フィールドboostでランキングし、フレーズ・ハイライト・ファセットで結果を提示し、品質を数値で測れる」状態。

### Day 7 — TF-IDFスコアリング

やること
- TF（文書内の語の頻度）とIDF（全文書での希少度）を計算し、文書スコアを算出。
- 検索結果をスコア降順でランキング表示。
- 文書数・DF（document frequency）などの統計をインデックスに保持。
- ポスティングのペイロードにTF（= 出現数）を格納（Day3で残した余白を使う）。

学習ポイント
- TF-IDFの直感（頻出 × 希少 = 重要）。
- 弱点の体感: 長文書が有利になりすぎる、TFが線形に効きすぎる。

チェックポイント
- [ ] 検索結果がスコア順に並ぶ
- [ ] IDFのために全文書数・DFを取得できる
- [ ] 意図的に長い文書を入れてスコアの偏りを観察できる

### Day 8 — BM25への拡張

やること
- TF-IDFをBM25に拡張。TF飽和（k1）と文書長正規化（b、avgdl使用）を導入。
- パラメータ k1（1.2前後）, b（0.75前後）を変えて結果の変化を観察・チューニング。
- 平均文書長 avgdl をインデックス統計として保持。

学習ポイント
- BM25がTF-IDFの弱点をどう補正するか（k1でTFを飽和、bで長さ補正）。
- これがElasticsearchのデフォルトスコアリングと同じ土俵であること。

チェックポイント
- [ ] BM25スコアで検索結果がランキングされる
- [ ] k1, b を変えたときの順位変化を再現・説明できる

## 補足: なぜこの構成か

v1（本線 = Boolean + BM25 + フレーズ）に対し、v2は**検索の種類を広げる**方向に振った。ワイルドカード・ファジーは「マッチするtermをまず集めてからポスティングをOR結合する」多段構造という点でBooleanの自然な拡張であり、ソート済みterm辞書（= LSMのソート済みSSTable）をprefix/近傍走査に再利用できる。ハイライトとファセットは、フレーズ検索で導入する位置情報とforward index（文書本体）の再利用で、既存資産を提示に転用する練習になる。

最後にprecision/recallの評価を置いたのは、検索種とパラメータ（ファジーのk、BM25のk1/b、boost）を増やすほど「本当に効いているか」の判断が主観的になるため。数値で測って調整するところまでを本線に含めることで、「機能を作った」ではなく「品質を測って良くした」まで到達する。分散処理・REST API層はElasticsearchを「製品」たらしめる部分だが、検索エンジンのコア原理の学習からは外れるので今回も対象外とした。
