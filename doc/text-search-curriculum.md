# 全文検索エンジン実装 — 2週間カリキュラム

Clojureで転置インデックスベースのミニ全文検索エンジンを実装し、Elasticsearch（Lucene）のコア原理を自分の手で再現する2週間の学習計画。

---

## 前提と方針

| 項目 | 決定 |
|---|---|
| 実装言語 | Clojure |
| トークナイズ | 日本語 + N-gram（bi-gram中心、自作） |
| 永続化層 | 前回実装したLSM-treeストレージエンジンを再利用 |
| スコアリング | TF-IDF → BM25 まで |
| 検索機能（本線） | AND/OR検索、フレーズ検索 |
| 検索機能（発展） | ワイルドカード、ファジー検索、ハイライト |
| データセット | 学習用の小規模サンプル（数百〜数千件）。実践投入（いしなび/これ飲も）は2週間後に別途検討 |

### 最終ゴール
「テキスト集合に対して、キーワード検索・BM25ランキング・フレーズ検索ができるミニ検索エンジン」をCLI/REPLから操作できる状態にする。これはElasticsearchのデフォルト検索（BM25 + bool query + match_phrase）の心臓部にあたるLucene相当の縮小版。

### Elasticsearchとの対応（作るもの / 作らないもの）

| 機能 | Elasticsearchでの呼称 | 今回 |
|---|---|---|
| AND/OR検索 | bool query (must/should) | ◎ 本線 |
| フレーズ検索 | match_phrase | ◎ 本線 |
| BM25スコアランキング | _score | ◎ 本線 |
| ワイルドカード | wildcard query | △ 発展 |
| ファジー検索 | fuzzy query | △ 発展 |
| ハイライト | highlight | △ 発展 |
| ファセット・集計 | aggregations | ✕ 対象外（別領域） |
| 分散・レプリケーション | sharding / replica | ✕ 対象外 |
| REST API層 | _search API | ✕ 対象外 |
| near real-time search | refresh | ✕ 対象外 |

## Week 1: トークナイズ 〜 転置インデックス 〜 Boolean検索

ゴール: 「サンプル文書をインデックス化し、AND/OR検索でヒット文書を返せる」状態。

### Day 1-2 — N-gramトークナイザの自作

やること
- bi-gram（2文字）を基本としたトークナイザを実装。`"日本酒"` → `["日本", "本酒"]`。
- 正規化処理（全角/半角統一、大文字小文字、記号除去、Unicode正規化NFKC）を前段に置く。
- uni-gram / bi-gram / tri-gram を切り替えられるようにし、テストで再現率と索引サイズのトレードオフを比較。

学習ポイント
- なぜ日本語で形態素解析ではなくN-gramを選ぶと「検索漏れが減るが誤ヒットが増える」のか（例: 「京都」で「東京都」がヒットする問題）。
- N数の選定が精度と索引サイズに与える影響。

チェックポイント
- [ ] 任意の日本語文字列をbi-gramトークン列に変換できる
- [ ] 正規化のユニットテストが通る（全角数字、濁点、記号など）
- [ ] N数を変えたときのトークン数・重複を計測できる

### Day 3-4 — 転置インデックスをオンメモリで構築

やること
- ポスティングリスト（term → 出現文書IDの一覧）のデータ構造を設計。
- 文書追加API（`add-document`）を実装。文書ID採番、term抽出、ポスティングへの追加。
- まずはClojureの永続データ構造（`sorted-map` + `sorted-set` など）でインメモリ実装。

学習ポイント
- 転置インデックスの構造（term辞書 + ポスティングリスト）。
- Clojureの永続データ構造・トランスデューサーがインデックス構築とどう噛み合うか。

チェックポイント
- [ ] 複数文書を追加し、任意のtermからポスティングリストを取得できる
- [ ] term辞書のサイズ、平均ポスティング長を確認できる

### Day 5 — LSM-treeへの統合設計

やること
- ポスティングリストをLSM-treeのkey/valueにどうマッピングするか設計。
  - 例: key = term、value = エンコードしたポスティングリスト（文書ID列 + 頻度 + 位置）。
- 文書IDのデルタエンコーディング + 可変長整数（varint）などで圧縮フォーマットを検討。
- インメモリ実装をLSM-tree永続化層に差し替え、書き込み→再起動→読み込みが通ることを確認。

学習ポイント
- LuceneのセグメントとLSM-treeのSSTableの発想の近さ（書き込み最適化 vs 読み込み最適化）。
- 「書いて、マージして、読む」構造が検索エンジンでも成り立つこと。

チェックポイント
- [ ] termをキーにポスティングをLSM-treeへ永続化できる
- [ ] プロセス再起動後もインデックスから検索できる
- [ ] ポスティングのエンコード/デコードのラウンドトリップテストが通る

### Day 6-7 — AND/OR（Boolean）検索

やること
- クエリをトークナイズし、各termのポスティングリストを取得。
- AND = 積集合、OR = 和集合をポスティングリストのマージで実装。
- ソート済みリストのマージで効率的に集合演算（skip前提の走査）。
- REPLまたは簡易CLIで検索できるようにする。

学習ポイント
- ポスティングリストの積集合・和集合アルゴリズム。
- ソート済みリストのマージがなぜ高速か。

チェックポイント
- [ ] `A AND B`、`A OR B` が正しいヒット文書集合を返す
- [ ] N-gramに分解された検索語が正しくAND結合される（例: 「日本酒」→「日本」AND「本酒」）
- [ ] Week1完了時点で「検索できる」状態

## Week 2: スコアリング 〜 ランキング 〜 発展機能

ゴール: 「BM25でランキングされた検索結果とフレーズ検索ができる」状態。

### Day 8-9 — TF-IDFスコアリング

やること
- TF（文書内の語の頻度）とIDF（全文書での希少度）を計算し、文書スコアを算出。
- 検索結果をスコア降順でランキング表示。
- 文書数・DF（document frequency）などの統計をインデックスに保持。

学習ポイント
- TF-IDFの直感（頻出 × 希少 = 重要）。
- 弱点の体感: 長文書が有利になりすぎる、TFが線形に効きすぎる。

チェックポイント
- [ ] 検索結果がスコア順に並ぶ
- [ ] IDFのために全文書数・DFを取得できる
- [ ] 意図的に長い文書を入れてスコアの偏りを観察できる

### Day 10-11 — BM25への拡張

やること
- TF-IDFをBM25に拡張。TF飽和（k1）と文書長正規化（b、avgdl使用）を導入。
- パラメータ k1（1.2前後）, b（0.75前後）を変えて結果の変化を観察・チューニング。
- 平均文書長 avgdl をインデックス統計として保持。

学習ポイント
- BM25がTF-IDFの弱点をどう補正するか（k1でTFを飽和、bで長さ補正）。
- これがElasticsearchのデフォルトスコアリングと同じ土俵であること。

チェックポイント
- [ ] BM25スコアで検索結果がランキングされる
- [ ] k1, b を変えたときの順位変化を再現・説明できる
- [ ] 同じクエリでTF-IDFとBM25の順位差を比較できる

### Day 12 — フレーズ検索（位置情報付きインデックス）

やること
- ポスティングリストに位置情報（term positions）を持たせる。
- フレーズクエリで、隣接する位置に出現する文書だけをヒットさせる。
- N-gramの連続性を利用した近接判定。

学習ポイント
- positional index（転置インデックスの一段深い構造）。
- これがワイルドカード・ハイライトの土台になること。

チェックポイント
- [ ] `"日本酒"` をフレーズとして扱い、語順・隣接を満たす文書のみ返す
- [ ] 位置情報のエンコード/デコードが永続化層で通る

### Day 13 — 最適化・ベンチマーク（+ 発展着手）

やること
- 検索速度、インデックスサイズ、構築時間を計測するベンチマークを用意。
- ポスティング走査の最適化（スキップリスト、ブロック単位読み出し）を検討・実装。
- 余裕があれば発展課題に着手:
  - ワイルドカード検索（前方一致索引）
  - ファジー検索（レーベンシュタイン距離）
  - ハイライト（位置情報の再利用）

チェックポイント
- [ ] 検索レイテンシとインデックスサイズを数値で出せる
- [ ] 最適化前後で性能差を計測できる

### Day 14 — 統合テスト・振り返り

やること
- サンプルデータセットでE2Eの統合テスト（インデックス構築 → 各種検索 → ランキング検証）。
- 学んだことの整理と、Elasticsearch/Luceneとの対応の振り返り。
- 実践投入の検討: 「いしなび」「これ飲も」への組み込み可否、必要な追加作業（実データ量でのスケール、更新頻度、既存Postgres全文検索との比較）を洗い出す。

チェックポイント
- [ ] 統合テストが一通り通る
- [ ] 2週間の成果を説明できる（設計判断の理由を言語化）
- [ ] 実践投入に向けた次アクションが明確

## 補足: なぜこの構成か

BM25まで到達すると「今のElasticsearchと同じスコアリング」を自作したことになる。フレーズ検索を本線に入れるのは、positional indexという転置インデックスの一段深い構造を学べ、発展機能（ワイルドカード・ハイライト）の土台にもなるため。ファセット検索や分散処理はElasticsearchを「製品」たらしめる部分だが、検索エンジンのコア原理の学習からは外れるので今回は対象外とした。
